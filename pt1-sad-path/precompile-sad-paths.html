Detailed Findings

All identified issues organized by severity tier. Click to expand for code references, mechanism details, and trigger conditions.

C1 Critical FP nondeterminism causes consensus divergence across hardware ONNX 0x0800 ▶
onnx_inference.rs:718-730

ONNX runs as a native synchronous precompile. IEEE 754 floating-point operations (especially FMA) produce platform-dependent results. AMD vs Intel CPUs will compute different least-significant bits, producing different state roots on different validators.

let session = Session::builder() .with_optimization_level(GraphOptimizationLevel::Disable) .with_intra_threads(1) // Not enough for cross-platform determinism .with_inter_threads(1) .commit_from_memory(&model_bytes)?;

The code attempts to enforce determinism by disabling optimizations and using single-threading, but this is insufficient for bit-exact reproducibility across different CPU architectures.

Impact: Chain fork between heterogeneous validators Why untested: All test nodes use identical GCP instance types
C2 Critical Floating-point sqrt in gas calculation ONNX Gas ▶
onnx_complexity.rs:428-438

Gas cost calculation uses flops.sqrt() and floating-point multiplication. Different CPUs produce different sqrt results, leading to different gas charges and different execution outcomes across nodes.

pub fn calculate_gas_cost(flops: f64, base_gas_cost: u64, flop_gas_rate: f64) -> u64 { let normalized_flops = flops.sqrt(); // NOT deterministic across CPUs let gas_from_flops = (normalized_flops * flop_gas_rate) as u64; base_gas_cost + gas_from_flops }
Impact: Different gas costs → different execution outcomes → consensus divergence
C3 Critical Chunk indexing panic on malformed tensor data ONNX 0x0800 ▶
types.rs:315-927 (multiple locations)

When a user sends a tensor where values.len() is not a multiple of the element size (e.g., 2 bytes for Float32 expecting 4-byte chunks), direct chunk[N] indexing panics with "index out of bounds". This is exactly the class of bug the ONNX incident exposed.

// to_vec_f32() - Panics if chunk.len() < 4 let val = f32::from_be_bytes([ chunk[0], chunk[1], chunk[2], chunk[3] // PANIC if incomplete ]);

Affected: to_vec_f32(), to_vec_f64(), to_vec_i32(), to_vec_i64(), to_vec_u32(), to_vec_u64(), fixed-point conversions, and RLP encoding (lines 816-927).

Trigger: Single malformed transaction Impact: Node crash
C4 Critical Stack overflow in JSON recursive traversal JQ 0x0803 ▶
jq.rs:181-214

compute_json_stats() uses pure recursion with NO depth limit. Deeply nested JSON (10K+ levels) overflows Rust's ~2MB stack. This happens before any gas or execution limits are checked.

fn rec(v: &Value, depth: u32, acc: &mut JsonStats) { acc.num_values = acc.num_values.saturating_add(1); acc.max_depth = acc.max_depth.max(depth); match v { // ...recursively calls rec() with depth + 1 // NO DEPTH LIMIT } }
Trigger: JSON with 10K+ nesting levels Impact: Process crash (stack overflow)
C5 Critical Stack overflow in array flattening JQ 0x0803 ▶
jq.rs:641-663

Recursive flattening with zero depth limit. Comment in code literally says "Always recurse". Deeply nested arrays [[[[...]]]] overflow the stack.

fn flatten_value(val: Value, output: &mut Vec<Value>) { match val { Value::Array(arr) => { for elem in arr { flatten_value(elem, output); // Always recurse - NO DEPTH LIMIT } } } }
Trigger: Nested arrays in JQ query result Impact: Process crash (stack overflow)
C6 Critical Buffer underflow panic in Nitro cert verification Nitro 0x0101 ▶
ritual-nsm/verifier.rs:144

Slices public key assuming it is at least 97 bytes. If a malformed certificate has a shorter key, the subtraction underflows and panics.

let public_key = &public_key[public_key.len() - 97..]; // If public_key.len() < 97: PANIC (attempt to subtract with overflow)
Trigger: Malformed X.509 certificate with small public key Impact: Node crash
C7 Critical Integer underflow in Nitro payload length calculation Nitro 0x0101 ▶
ritual-nsm/verifier.rs:154

For any payload under 4352 bytes, payload.len() + 94 - 4446 underflows. In debug mode this panics; in release mode it wraps, producing incorrect verification data.

let payload_length_bytes: u8 = (payload.len() + 94 - 4446) .try_into() .map_err(|err| VerificationError::PayloadLengthBytesConversionFailed(err))?; // payload of 1000 bytes: 1000 + 94 - 4446 = -3352 → UNDERFLOW
Trigger: Payload < 4352 bytes Impact: Panic (debug) or incorrect verification (release)
C8 Critical Shape product overflow causes 17GB allocation attempt ONNX 0x0800 ▶
types.rs:325, 357, 381, 396

shape.iter().product() on user-supplied dimensions can overflow. Shape [65535, 65535] with Float32 requests ~17GB allocation, causing OOM crash. No validation of shape product before allocation.

Trigger: Large shape dimensions in tensor Impact: OOM crash
C9 Critical Atomicity violation: async execution without settlement Async FSM ▶
ethereum/payload/src/lib.rs

If an async transaction executes successfully but the settlement doesn't fit in the block, the execution succeeds but settlement is skipped without rollback. The executor performs work but is never paid. This is a known economic attack vector that remains unfixed.

Status: KNOWN, UNFIXED Impact: Executor griefing attack
H1 High context.Background() causes infinite hang in LLM handler Executor ▶
llmcall/handler.go:386, audiocall/handler.go:323

Attestation verification uses context.Background() inside a singleflight call. If vLLM proxy is unreachable, the first request hangs forever, and all subsequent requests queue behind it in the singleflight group.

Impact: All LLM/audio requests hang indefinitely
H2 High JQ timeout only checked between output iterations JQ 0x0803 ▶
jq.rs:460-463

Timeout is checked after outputs.next() returns. If a JQ filter blocks on a single computation that never yields, the timeout is never evaluated. Can hang the node thread.

Trigger: JQ expression with single expensive operation
H3 High No size limit on JSON input before parsing JQ 0x0803 ▶
jq.rs:591-598

serde_json parses arbitrarily large JSON before any gas or execution limits are applied. A 100MB+ JSON input causes OOM with ~2x memory overhead during parsing.

Trigger: Very large JSON input in JQ call Impact: OOM crash
H4 High Unbounded PCR map and cabundle allocation in Nitro Nitro 0x0101 ▶
ritual-nsm/verifier.rs:373-427

No limit on PCR map entry count or cabundle array size. Attacker can send CBOR with 1M PCR entries, causing unbounded memory allocation. Fixed 100K gas regardless of input size.

Trigger: Large CBOR attestation document Impact: Memory exhaustion
H5 High Fixed gas costs enable CPU exhaustion attacks Ed25519 / SECP256R1 / Nitro ▶
sigver.rs:26-30

All crypto precompiles charge fixed gas (Ed25519: 2,000, SECP256R1: 3,450, Nitro: 100,000) regardless of input size. 100MB message at 2,000 gas makes verification extremely cheap for attackers.

Impact: CPU exhaustion at low gas cost
H6 High SSE scanner 64KB default buffer limit Executor LLM ▶
llmcall/handler.go:1477

Go's bufio.Scanner default max buffer is 64KB. If a single SSE line from vLLM exceeds this (e.g., large function call output), the scanner errors with "token too long" and the stream breaks.

Impact: Streaming responses silently truncated
H7 High Parser recursion in cost analysis bypasses gas limits JQ 0x0803 ▶
jq_cost.rs:834-857

The cost analysis parser itself has no recursion depth limit. Pathological JQ like map(map(map(...))) with 150+ levels causes stack overflow during analysis, before gas estimation even runs.

Impact: Crash before gas check
H8 High io.ReadAll without size limit on agent orchestrator response Executor Agent ▶
agentcall/handler.go:430

io.ReadAll(resp.Body) reads the entire orchestrator response without size limits. A malfunctioning or adversarial orchestrator returning a multi-GB response causes OOM in the executor.

Impact: Executor OOM
F1 FSM Nested async from delivery callback untested Async FSM ▶

When a Phase 2 delivery settlement's callback calls another async precompile, a nested commitment is created. Fee calculation must use the nested job's commit_block, not the original. This multi-stage path requires complex simulation and has no test coverage.

Risk: Builder/verifier divergence on nested fee calculation
F2 FSM Settlement for expired job: builder/verifier asymmetry Async FSM ▶

Builder gracefully skips expired settlements (logs + continues). Verifier hard-fails with BlockVerificationError. If timing differs between them (e.g., block boundary race), the verifier rejects a block the builder considered valid.

Risk: Block rejection under timing edge cases
F3 FSM Settlement for non-existent job race condition Async FSM ▶

Race between job creation (commitment included in block) and settlement arrival. If settlement references a job that hasn't been committed yet, builder skips it but verifier may hard-fail depending on state.

Risk: Block rejection during high-throughput periods
F4 FSM Phase 2 deadline extension interaction untested Async FSM ▶

Expiry calculation uses max(original_expiry, phase2Deadline). The interaction between original TTL and Phase 2 deadline extension has no test coverage, particularly when Phase 2 deadline is shorter than original TTL.

F5 FSM Fee calculation overflow with max values Async FSM ▶

Integer overflow potential in fee distribution when max gas values or extreme block distances are used. Edge case in fee struct construction where executor_fee + commitment_fee + inclusion_fee could overflow u256.

F6 FSM Multiple concurrent settlements atomicity under load Async FSM ▶

When multiple FulfilledAsync + Settlement pairs exist in a single block, if one settlement fails mid-execution, the atomicity guarantee (skip both fulfilled + settlement) needs to not affect neighboring pairs. No load testing validates this.

Floating-Point Consensus Risk

The single most critical architectural issue. Native ONNX precompile produces nondeterministic results across CPU architectures, creating a consensus failure vector.

⚠ Architecture-Level Consensus Risk

ONNX (0x0800) is a synchronous native precompile — inference runs inside block execution and results go directly into EVM state. IEEE 754 floating-point is not bit-exact across hardware. Different validators will compute different state roots, causing chain forks.

Step 1
Node A (Intel) runs ONNX inference, gets result r_A
Step 2
Node B (AMD) runs same inference, gets r_B ≠ r_A (FMA difference)
Step 3
Node A builds block with state root from r_A
Step 4
Node B re-executes, computes different state root
Step 5
Node B rejects the block → Chain Fork
