# Delegated Secrets Auth Redesign

## What The Signing Claims Mean

There are two distinct keys in the delegated-secrets encryption flow:

- The **owner key** is the durable EOA or passkey identity that maps to an address used by access-control policy.
- The **ephemeral ECIES key** is a fresh per-ciphertext key generated by the client-side encryption library and embedded in the encrypted blob as an ephemeral public key.

An owner-key signature proves:

```text
The owner identity for this Ethereum address authorizes the signed statement (e.g. "I own ethereum address X and I claim Y is true").
```

An ephemeral-key signature proves:

```text
The party that generated the ciphertext authorizes the signed statement (e.g. I generated the secret and I claim Y is true)
```

So what: we may want bidirectional claims. E.g. we may want automated checks for the statements "I own ethereum address X and I approve this is my secret" as well as "I generated the secret and I approve address X to use it."

## Why Both Claims Matter

The **ephemeral-key claim is the higher-severity control** because it binds the ciphertext to the owner label. Without it, a public ciphertext can be copied and re-labeled by a different party. This is the `origin/dev` re-sign bypass: an attacker observes Alice's `encrypted_secrets`, signs the unchanged ciphertext as Bob, submits it as Bob, hits the self-owned fast path, and the executor decrypts Alice's blob into Bob's request.

The **owner-key claim is useful for attribution and future policy safety** because it prevents arbitrary ciphertexts from entering Alice's ownership namespace without Alice's approval. This is less directly severe than the re-sign bypass when grants are strictly per-secret-hash, but it still matters:

- **Audit/logging:** systems should not report "Alice's secret was used" unless Alice approved that ownership claim.
- **Broad policy risk:** if any current or future grant becomes owner-wide rather than hash-specific, false Alice ownership could unlock unintended access.
- **Confused deputy risk:** future features may use owner identity for preferences, storage namespaces, allowlists, quotas, or enforcement paths; Bob should not be able to trigger Alice-scoped behavior by labeling his ciphertext as Alice-owned.

## Arshan PR Pattern: Owner Inside Ciphertext

Arshan's PR has an equivalent of the ephemeral-key owner-binding claim by putting `secret_owner` inside the AES-GCM-protected plaintext envelope. This means the ciphertext itself decrypts to "I am owned by Alice," and byte-level tampering with that owner field should fail GCM tag verification.

This is equivalent in the security property that matters most for the re-sign bypass: Bob cannot copy Alice's ciphertext, set the public owner field to Bob, and still pass the executor's decrypted-owner check. The ciphertext remains internally bound to Alice.

Pros versus a separate ephemeral signature:

- **Simpler request shape:** no extra signature field, no new signature format, and no circular hashing problem.
- **Uses existing AEAD integrity:** AES-GCM already authenticates the encrypted owner field against ciphertext tampering.
- **Directly fixes the observed bypass:** copied ciphertexts carry their original owner label through decryption.

Cons versus a separate ephemeral signature:

- **Less intuitive protocol model:** the envelope pattern is implementation-simple, but the separate ephemeral signature is clearer at the protocol layer because it directly represents the claim "the ciphertext-generating key says owner is X."
- **Post-decrypt check:** the executor must decrypt before learning the internal owner claim; a separate ephemeral signature can be checked before plaintext is materialized.
- **Owner is hidden even though it is operationally public:** if the owner should be clear metadata, putting it only inside ciphertext is slightly awkward and duplicates the public `secret_owners[i]` field.

Net: the PR pattern is a pragmatic equivalent for the high-severity re-sign bypass, assuming every decrypt path verifies the encrypted owner against the public expected owner. A separate ephemeral signature is more intuitive and cleaner as protocol transcript design, but it adds complexity and probably should be justified by a need for pre-decrypt verification, public auditability, or long-term protocol clarity.
