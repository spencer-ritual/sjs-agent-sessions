## Summit-Specific Review Notes

| Priority | Issue | Notes |
| --- | --- | --- |
| Critical | Fix the validator-registry update path before treating StakeHub membership as dynamic. | The current draft appears effectively fixed to the launch validator registry unless off-chain tooling mutates and redistributes `/config/validator_registry.toml`. Summit reads the initial registry path from `SUMMIT_VALIDATOR_REGISTRY_PATH` and reloads that file, but Summit itself does not write or update it. If StakeHub elects a validator whose consensus address is not already present in the local registry, `stakehub_committee_for_block` returns `None` and Summit retains the existing committee. That means new on-chain StakeHub membership can fail open to the original set, so dynamic validator admission is registry-gated rather than purely StakeHub-driven. |
| High | Make the StakeHub committee recovery FSM explicit. | If the StakeHub read fails, returns empty, or contains a validator missing from Summit's BLS registry, Summit falls back to the current active committee. That is good liveness thinking, but the recovery state machine is implicit: the code logs and retains the previous committee, then retries at the next epoch boundary. The PR should make the intended recovery behavior explicit, at least in documentation: when a node is allowed to carry the old committee forward, how operators are expected to repair the missing BLS registry entry or StakeHub read path, whether this should be observable/alerting-critical, and whether there is a bounded retry or fail-closed condition after repeated epochs. |
| High | Map StakeHub committee members by the actual consensus address, not `fee_recipient`. | The StakeHub ABI returns `consensusAddrs`, while Summit looks committee members up through metadata keyed by `ValidatorAccount.fee_recipient`. If those fields differ, StakeHub committee updates can be dropped and Summit can retain the old committee. Add an explicit consensus-address field/mapping, or query/derive the operator mapping consistently. |
| High | Re-check the light-client / epoch-certificate verification property. | A property we had before is that a verifier could follow the chain by verifying epoch certificates, because validator updates happened only at epoch boundaries and the old validator set signed the transition to the new validator set. Before this PR, the next validator set was explicit in Summit-owned transition data: the epoch-boundary block carried `added_validators` / `removed_validators`, and a verifier could update the validator set from the old Summit state plus those signed transition fields. With this PR, the next epoch validators are not explicit in Summit's epoch-transition fields in the same way. The block still has fields like `added_validators` / `removed_validators`, but the legacy Summit FSM that populates/applies those is effectively removed/nooped for validator membership. The actual next committee is read after finalization from EL `RitualStakeHub` at the boundary block hash, while the on-chain `OnchainConsensusOutcome` currently passes `quorum_cert: None`. The boundary certificate still proves the old committee finalized the boundary block, and the boundary header commits to the checkpoint/snapshot digest, but the next validator set is not obviously proven by "epoch certificates only" unless the verifier also validates or trusts the EL StakeHub state/snapshot that produced the next committee. This needs an explicit design answer. |
| High | Define how StakeHub validators get authorized Summit BLS public keys. | The old Summit deposit path required both the node identity key and the BLS consensus key to sign the deposit message, so Summit had evidence that a validator entering the CL committee controlled the keys it registered. In the new path, `RitualStakeHub` supplies a consensus/operator address and voting power, while Summit maps that address to `(node_public_key, consensus_public_key)` from local Summit metadata. In this PR, that metadata appears to come from the genesis/validator registry path (`SUMMIT_VALIDATOR_REGISTRY_PATH`) or an EL snapshot during bootstrap, not directly from StakeHub. The PR should document and test the authority path for new validators: how a StakeHub entrant publishes its Summit node key and BLS key, who validates those keys, how the registry is distributed to all Summit nodes, and what prevents an address/key mismatch from silently keeping the old committee. |
| High | Make the StakeHub registration to Summit peering flow explicit. | The implied local flow is: run `configure-node` to generate Summit Ed25519/BLS keys, append the validator to `validator_registry.toml`, and copy `bootstrappers.toml`; run `register-node` to call `RitualStakeHub.createValidator`; then run `start-node` so the node peers through Summit bootstrappers and eventually enters once StakeHub elects it and the existing nodes can map its StakeHub address to Summit keys. Before this PR, BLS/node-key validation was coupled to CL committee entry through the deposit request signatures, while P2P reachability still depended on Summit address/bootstrap metadata. In the new flow, StakeHub registration, BLS registry distribution, and network peering are separate steps. That may be the right architecture, but all of it should be explicit: whether a validator must peer before on-chain registration, whether it can register before existing nodes know its BLS/P2P metadata, how all validators receive the updated registry, and what failure mode is expected if the node is elected on-chain but not yet discoverable or not yet in the local registry. |
| High | Restore equivalent recovery/checkpoint coverage for StakeHub-driven validator-set changes. | The PR deletes the legacy execution-request test suite and several harnesses around staking, checkpointing, outdated checkpoints, withdrawals/exits, and sync-from-genesis after validator-set changes. Some old FSM tests are obsolete because StakeHub now owns staking state, but the recovery scenarios are still essential. Add replacement tests that prove a node can checkpoint/bootstrap/sync across StakeHub-driven committee changes, including after a validator joins or exits, with the BLS registry/snapshot metadata preserved. This should include manual e2e tests, not just unit tests, because the failure mode spans Summit, Reth, EL snapshots, and StakeHub reads. |
| Medium | Document the default epoch duration and choose a production-scale value. | Summit defaults to `DEFAULT_BLOCKS_PER_EPOCH = 100`, unless `SUMMIT_BLOCKS_PER_EPOCH` is set. At 300ms/block, the default epoch is `100 * 0.3s = 30s`, which is `30 / 86_400 = 0.000347222...` days. That is very short for validator-set rotation: there is no universal blockchain standard, but production epoch/validator-set intervals are commonly measured in minutes to days, not seconds. At 300ms/block, 1 hour implies `12_000` blocks (`0.041666...` days), 1 day implies `288_000` blocks, and 1 week implies `2_016_000` blocks (`7` days). Longer epochs reduce committee churn, StakeHub read pressure, fallback exposure, and light-client checkpoint frequency, at the cost of slower staking changes. |
| Medium | Document the intentional round-robin fallback for equal or missing weights. | The orchestrator explicitly uses round-robin leader election when `stake_weights.is_empty()` or every stake weight equals the first weight. Otherwise it constructs `StakeWeightedConfig::new(stake_weights)`. The likely rationale is reasonable: if all weights are equal, stake-weighted leader selection is behaviorally equivalent to uniform selection, and round-robin is simpler, deterministic, and legacy-compatible. The concern is that the PR does not say this. This looks like an agent-picked fallback rather than an explicitly documented protocol decision, and this codebase should prefer explicit decisions for consensus behavior. Please either document the rationale in code/PR notes or route equal weights through the same stake-weighted elector if that is the intended protocol surface. |
| Medium | Correctly tightens EL commit/forkchoice selection, but should document the rationale explicitly. | Before this PR, Summit had looser EL acceptance behavior: `execute_block` did not return a boolean, and there was a path where `payload_status.is_syncing()` still advanced `state.forkchoice.head_block_hash`. Invalid blocks could still be "kept as part of chain" from the CL state perspective after logging validation failure. Now, `execute_block` is a gate: a block only advances local consensus state if EL says the payload is valid, the block withdrawals match the expected CL withdrawals, and the block's EL parent matches Summit's current forkchoice head. If any of those fail, `execute_block` returns `false`. That return value is then used in two important places: for notarized blocks, Summit does not store the fork state and does not commit the fork to Reth if EL rejected the block; for finalized blocks, Summit refuses to advance canonical state if EL rejected the block. The likely rationale is correct: once StakeHub becomes the source of truth for validator membership, voting power, and rewards, Summit cannot safely finalize or build on top of CL blocks whose EL state did not execute. The next committee is read from EL `RitualStakeHub` at the epoch boundary. If Summit advanced over an invalid, syncing, or parent/withdrawal-mismatched EL payload, then CL could believe it moved to a new height or epoch while EL did not have the state needed to derive the same committee. This is a good source-of-truth alignment, but the PR should document it explicitly as consensus/EL agreement behavior rather than leaving it as an incidental implementation change. |
| Medium | Remove or replace the legacy DepositContract bootstrap path. | `RethEngineClient::get_deposit_minimum_state`, `deposit_contract_address`, and `apply_deposit_minimum_state` still read `minimumDepositState()` from the retired deposit contract address while the sibling contract PR deletes `DepositContract.sol` and moves staking to `RitualStakeHub`. Replace this with the StakeHub minimum/self-delegation source or delete the dead bootstrap path. |
| Low | Map old Summit validator-lifecycle settings to the new StakeHub/BSC-derived parameters. | Several behavior settings likely changed as a side effect of replacing the Summit validator FSM with `RitualStakeHub`: delayed activation and exit semantics, refund/rejection behavior for malformed or inconsistent validator operations, min/max stake handling, partial-withdrawal-to-full-exit behavior, and rate limiting / bounded churn such as the old `VALIDATOR_ONBOARDING_LIMIT_PER_BLOCK`. It may be correct to prefer the Binance-derived StakeHub parameters and semantics, but the PR should include an explicit mapping from old Summit behavior to new StakeHub behavior so reviewers can distinguish intentional protocol changes from accidental removals. |

## Code Pointers

- Default epoch length: `node/src/engine.rs` defines `DEFAULT_BLOCKS_PER_EPOCH = 100` and reads `SUMMIT_BLOCKS_PER_EPOCH`.
- Validator registry path: `SUMMIT_VALIDATOR_REGISTRY_PATH=/config/validator_registry.toml`.
- StakeHub committee read and fallback: `finalizer/src/actor.rs::stakehub_committee_for_block`.
- Next-epoch committee selection: `finalizer/src/actor.rs` uses `stakehub_committee_for_block(&block)` at epoch transition, else falls back to current active validators.
- Legacy deposit bootstrap: `types/src/engine_client.rs` still exposes deposit minimum state helpers.
- Leader election fallback: `orchestrator/src/actor.rs::enter_epoch` uses round-robin when weights are empty or all equal.
- Strict EL acceptance: `finalizer/src/actor.rs::execute_block` returns `false` unless payload status, withdrawals, and parent match.
- Snapshot digest: `finalizer/src/actor.rs` builds `OnchainConsensusOutcome` at epoch boundaries with `quorum_cert: None`.
