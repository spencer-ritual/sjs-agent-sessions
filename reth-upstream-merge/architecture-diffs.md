# Architecture Diffs

This file captures high-signal architectural differences between upstream-merge implementations. The goal is to record where behavior moved, what the alternative design is, and what should be verified before choosing one approach.

Scope note: this comparison now includes both `ritual-reth-internal` and `ritual-reth-nodebuilder-internal`. The latest checked Arshan tips were Reth `1c28021` and nodebuilder `2536ce1`; the latest checked Spencer nodebuilder tip was `2a51426`.

## Implementation Diffs vs Old `dev`

- **Sequencing rights: Spencer extracts payload-time sequencing-rights ordering into a dedicated payload module.**
  - In old `dev`, sequencing rights were already part of payload construction. The old payload flow collected candidate transactions, sorted by effective tip, classified transactions for sequencing rights, called `ritual_sequencing_rights::reorder_block(...)`, then reordered the payload collection before execution.
  - Spencer preserves the broad placement decision that sequencing rights are a payload-time ordering concern, but changes the shape: he extracts the ordering helper into `crates/ethereum/payload/src/sequencing_rights.rs` and applies it to `PayloadCandidate`s from the new v2.2 payload loop.
  - By "new v2.2 payload loop", we mean the upstream Reth v2.2 payload-builder shape centered on `EthereumPayloadBuilder` and installed through the node builder component stack as `EthereumNode::components().payload(BasicPayloadServiceBuilder<...>)`. The upstream extension point is the `PayloadBuilderBuilder` trait: a node can keep the rest of `EthereumNode::components()` and replace the payload builder with a custom one, rather than editing txpool internals or scattering block-construction policy across unrelated modules.
  - Spencer aligns with this pattern at the **internal payload-loop level**: he ports sequencing rights into the v2.2 payload candidate flow instead of preserving the old `dev` unified-builder files. That is better aligned with Reth v2.2 than keeping the old payload architecture alive.
  - Spencer does **not** fully align with the cleanest **external nodebuilder pattern** yet: the sequencing-rights code still lives inside the Reth repo's `crates/ethereum/payload` implementation rather than in `ritual-reth-nodebuilder-internal` as a custom `PayloadBuilderBuilder` / custom payload component. The cleaner end state may be to move this extracted helper into a Ritual payload-builder component in nodebuilder, leaving upstream Reth's Ethereum payload builder closer to stock.
  - The alternative is to keep sequencing-rights ordering embedded directly in the main payload builder / unified execution flow, as old `dev` did. That keeps all block-building phases in one control flow, but makes the payload builder larger and harder to merge. Spencer's extraction is cleaner architecturally if the helper stays small and has focused tests.
  - Important review question: old `dev` had an optional full-call-graph classification path using `ContractCallInspector` gated by `RITUAL_SEQUENCING_RIGHTS_FULL_CALL_GRAPH_ENABLED`. Spencer's payload reorder path appears to use `try_classify_static(...)` or `classify_top_level_with_sr(...)`; verify whether dropping payload-time full-call-graph simulation is intentional and behaviorally acceptable.
- **Scheduled transactions: Arshan preserves the old txpool-centered mapping and consumes it from nodebuilder; Spencer reimplements scheduled inclusion without a dedicated scheduled txpool.**
  - In old `dev`, scheduled transactions had explicit txpool state. The transaction pool had a dedicated scheduled subpool (`ScheduledPool`), a `SubPool::Scheduled` state, a `ValidPoolTransaction::new_scheduled(...)` constructor, and accessors such as `get_scheduled_transactions(...)`, scheduled removal, and scheduled expiry cleanup. This made scheduled transactions look like a special txpool-resident class that the payload builder could consume.
  - Arshan largely preserves that old mapping. His branch still has `crates/transaction-pool/src/pool/scheduled.rs`, `SubPool::Scheduled`, scheduled txpool accessors, and `new_scheduled(...)`-style construction. That is closer to old `dev` and easier to compare mechanically against the old code.
  - Arshan's latest nodebuilder work then consumes that state from `crates/node/src/payload.rs` via a custom `RitualPayloadBuilder` installed through `BasicPayloadServiceBuilder<RitualPayloadBuilderBuilder>`. Scheduled selection is not in stock Reth's `crates/ethereum/payload`; it is in nodebuilder. The builder reads `pool.scheduled_transactions()`, computes `pool_coverage`, scans or derives Scheduler obligations with `compute_obligations_from_pool(...)` / `compute_obligations(...)`, predicate-filters obligations, creates missing scheduled system txs with `augment_from_obligations(...)`, filters pool scheduled txs, applies heartbeat revival-lane selection, and combines `ScheduledPayloadCandidate::Pool` with `ScheduledPayloadCandidate::Augmented`.
  - Spencer deletes the dedicated scheduled txpool module and removes scheduled transactions as a txpool subpool. Instead, scheduled behavior is reimplemented in the payload/block-building path: payload code tracks `included_scheduled_indices`, extracts scheduler-routed scheduled tx info, synthesizes heartbeat/revival scheduled transactions, verifies scheduled constraints via `verify_scheduled_transactions_full_with_predicate_evaluator(...)`, and skips pooled scheduled duplicates that were already fulfilled from Scheduler state.
  - **Node-builder alignment:** Paradigm Reth v2.2 already has txpool subpools (`Pending`, `Queued`, `BaseFee`, `Blob`), but they are an internal validity/fee/nonce state machine rather than a generic custom-lane registry. If scheduled txs are normal mempool inventory that should have txpool RPC visibility, removal, limits, listeners, replacement, and propagation semantics, then `SubPool::Scheduled` is a fork-level txpool change. The cleaner node-builder version would be a custom `PoolBuilder` that supplies a custom `TransactionPool`, probably wrapping stock `EthTransactionPool` plus scheduled storage, but that is a large contract to maintain.
  - If scheduled txs are block-construction/system candidates derived from Scheduler state, then the more Reth-native node-builder integration is a custom payload component via `PayloadBuilderBuilder` / `BasicPayloadServiceBuilder`. Arshan now does this in nodebuilder while still preserving the old scheduled subpool as one input source. Spencer does the block-construction placement more directly by avoiding the dedicated scheduled subpool.
  - The architectural tradeoff is now sharper: Arshan has better behavioral continuity with old `dev` because both scheduled-pool inventory and Scheduler-derived augmentation feed the nodebuilder payload builder, but he still owns a fork-level txpool extension. Spencer is closer to stock txpool architecture, but must prove that removing txpool ownership preserves old observability/removal/duplicate semantics.
  - Important review question: removing `ScheduledPool` changes ownership of admission bypass, expiry/removal, RPC or pool observability, duplicate handling, and C1-C11 scheduled validation. Spencer's payload path clearly handles inclusion/verification pieces, but needs targeted probes to prove the old txpool-visible behaviors are either preserved elsewhere or intentionally removed.
- **Async lifecycle: Arshan preserves more txpool/verifier state; Spencer moves more async settlement construction into payload.**
  - In old `dev`, async lifecycle was split across txpool and payload. Txpool owned local lifecycle state such as `AwaitingExecution`, `Fulfilled`, `AwaitingDelivery`, and `DeliveryReady`; payload construction consumed that state through calls like `get_fulfilled_original_async_commitment_transactions(...)` and `get_delivery_ready_async_transactions(...)`, then built and executed settlement transactions in the block-building path.
  - Arshan preserves the txpool/verifier side of the old mapping in Reth and, in the latest nodebuilder branch, also restores a custom payload-builder path. The nodebuilder `RitualPayloadBuilder` contains async context reset, async commitment execution, settlement creation/execution, fulfilled-entry handling, delivery-ready handling, and mixed payload execution. This corrects the earlier Reth-only read: Arshan moved payload-side async construction into nodebuilder rather than stock Reth payload.
  - Spencer diverges more structurally inside Reth itself: his txpool still tracks async metadata, but fulfilled and delivery-ready entries are consumed by `crates/ethereum/payload/src/lib.rs`, with settlement preparation and execution factored into `crates/ethereum/payload/src/ritual_async.rs`.
  - **Reth v2.2 alignment:** Arshan is now closer to the clean nodebuilder pattern because Ritual async payload behavior lives in a custom payload component and stock Reth payload can stay closer to upstream. Spencer is closer to the v2.2 internal payload-loop shape inside Reth, but less cleanly separated from upstream Reth.
  - **Node-builder alignment:** Arshan's latest shape is the stronger nodebuilder extraction direction: `RitualNode` installs `RitualPayloadBuilderBuilder`, `RitualPoolBuilder`, and Ritual EVM/config components. Spencer's conceptual placement is right, but the code still lives more inside the Reth payload crate.
  - Important review question: run targeted probes for the async lifecycle: canonical commitment creates an awaiting-execution entry; executor result makes it fulfilled; payload includes the original plus Phase 1 settlement; long-running Phase 1 moves to awaiting-delivery; delivery result makes it delivery-ready; payload includes Phase 2 delivery settlement; canonicalization removes the right async entry without unlocking the sender too early.

## Feature Preservation Map Comparison

- **1. Transaction formats and codecs: Arshan centralizes Ritual primitive wrappers; Spencer ports more upstream-shaped storage/RPC codecs.**
  - Old `dev` carried custom transaction families directly through Ethereum primitives, storage codecs, receipts, and RPC conversion. Spencer keeps more of that file-level shape in upstream-style locations such as `crates/storage/codecs/src/alloy/transaction/{scheduled,async_commitment,async_settlement,passkey}.rs` and receipt adapters. Arshan moves more of the custom primitive surface under `crates/ritual/primitives`, which is cleaner as a Ritual namespace but makes storage/RPC adapter coverage harder to read from the normal Reth codec paths.
  - **Reth v2.2 alignment:** neither approach can avoid touching transaction identity surfaces; custom tx types are below node-builder. Spencer is closer to the upstream storage-codec extension points, while Arshan is closer to a separated Ritual primitive facade.
  - **Node-builder alignment:** node-builder cannot paper over transaction envelope and DB identity. The future-clean shape is a small, audited Ritual primitive/codec layer with explicit adapters into Reth storage, RPC, receipt, and CL/EL conversion.
  - Important review question: round-trip every custom tx type through RLP/2718, compact codec, DB/static-file, receipt, and RPC paths, including `spc_calls`, `delivery_spc_calls`, passkey signatures, and removed async fields such as `target_block` / `commit_block`.
- **2. Async precompile detection and validation: Arshan keeps shared validation and wires it through nodebuilder; Spencer wires it through Reth payload.**
  - Old `dev` had shared async detection/validation plus payload and verifier re-simulation. Arshan keeps a fuller `crates/ritual/async` validation surface and RPC validator, and latest nodebuilder consumes it from `RitualPayloadBuilder`. Spencer has less standalone detector surface, but uses async detection in the Reth v2.2 payload path via `AsyncContextFactory`, `SharedAsyncPrecompileInspector`, and `build_short_async_commitment_transaction(...)`.
  - **Reth v2.2 alignment:** shared validation belongs outside payload, but the authoritative detection path still has to be invoked by payload and verifier. Both now have a payload invocation path; Arshan's is externalized into nodebuilder, Spencer's is embedded in Reth payload.
  - **Node-builder alignment:** the best extraction is a Ritual async validation crate consumed by RPC, verifier, and a custom payload builder. Arshan is now closer to that shape; Spencer still has a cleaner internal payload-loop port but less separation.
  - Important review question: prove RPC rejection, payload detection, and verifier pre-execution simulation agree on TTL, Phase 2 deadline, executor capability, wallet lock, active-job, and malformed input cases.
- **3. EVM context, precompiles, and SPC: Arshan is pushing context into a Ritual EVM layer; Spencer preserves more SPC/precompile support crates and payload usage.**
  - Old `dev` threaded async/SPC context through EVM execution, precompile dispatch, payload building, and verification. Arshan's latest work adds verification-context plumbing under `crates/ritual/evm`, which is a good separation direction, but the marker surface for SPC/precompile support is thinner in the refreshed branch. Spencer keeps more flat support crates such as `ritual-spc-verification`, `ritual-precompile-addresses`, and `ritual-precompile-codecs`, with payload/verifier call sites.
  - **Reth v2.2 alignment:** the important upstream fit is the `ConfigureEvm` / EVM factory boundary and revm context shape. Direct edits to generic upstream EVM/precompile dispatch increase future revm/Reth merge cost.
  - **Node-builder alignment:** a clean Ritual node should provide a custom EVM/executor component that installs async-aware precompiles and context, rather than requiring unrelated upstream Ethereum EVM code to know Ritual policy.
  - Important review question: run one real payload-build, import/verifier, and RPC-simulation path for an async precompile and confirm all three use async-aware precompile dispatch, not the plain non-context precompile map.
- **4. Txpool async and scheduled lifecycle: Arshan stays closer to old txpool state; Spencer reduces custom scheduled txpool state but keeps async metadata for payload.**
  - Old `dev` used txpool as a major lifecycle owner for both scheduled and async work. Arshan preserves `ScheduledPool`, `SubPool::Scheduled`, async metadata RPC access, and the broad async FSM. Spencer removes scheduled as a txpool subpool, keeps an async pool, and shifts scheduled/fulfilled/delivery consumption toward payload.
  - **Reth v2.2 alignment:** upstream `SubPool` is a fixed validity/fee/nonce state machine, not a custom-lane registry. Arshan's scheduled txpool is a fork-level extension; Spencer's scheduled path is closer to Reth's payload-candidate model.
  - **Node-builder alignment:** if a feature needs mempool inventory semantics, use a custom `PoolBuilder` and custom `TransactionPool`; if it is block-construction/system work, prefer a custom payload builder.
  - Important review question: run the scheduled and async FSM probes for insertion, expiry, sender locks, canonical cleanup, reorg/backfill, duplicate handling, and txpool RPC visibility.
- **5. Scheduling and heartbeat: Arshan moves scheduled/heartbeat payload behavior into nodebuilder; Spencer embeds more of it in Reth payload.**
  - Old `dev` combined Scheduler hydration, due-call selection, C1-C11 verification, heartbeat revival, and payload augmentation. Arshan keeps scheduling code under `crates/ritual/scheduling` and verification under `crates/ritual/block-verification`, and latest nodebuilder `RitualPayloadBuilder` computes obligations, predicate-filters them, augments missing scheduled transactions, and selects heartbeat revival-lane candidates. Spencer carries `ritual-scheduled-verification`, `ritual-scheduled-block-building`, heartbeat tracking, and Reth payload calls such as `synthesize_heartbeat_revival_transactions(...)`.
  - **Reth v2.2 alignment:** due-call selection and heartbeat revival are payload-construction responsibilities with verifier replay, not normal txpool validity rules. Both implement that placement; Arshan does it through nodebuilder, Spencer through Reth payload.
  - **Node-builder alignment:** this wants a custom Ritual payload component plus shared scheduled-verification support. Arshan is now closer to the desired externalized component shape; Spencer is closer to an in-Reth port of old behavior.
  - Important review question: compare blocks with one due scheduled tx, one false predicate, one expired call, one active-async exemption, and one heartbeat revival.
- **6. Payload building: Arshan preserves upstream Reth payload by extracting a Ritual payload builder into nodebuilder; Spencer preserves more old Ritual behavior inside Reth payload.**
  - Old `dev` had a large custom payload builder that collected regular, scheduled, fulfilled async, delivery, and sequencing-rights candidates. Arshan's `crates/ethereum/payload/src/lib.rs` remains much closer to stock Reth because the custom work moved to nodebuilder `crates/node/src/payload.rs`. That builder now handles scheduled pool candidates, Scheduler-derived augmentation, predicate filtering, heartbeat revival selection, async commitment execution, fulfilled/delivery settlements, sequencing-rights classification, mixed ordering, metrics, and cleanup boundaries. Spencer's Reth payload is much larger and consumes those Ritual candidates directly inside `crates/ethereum/payload`.
  - **Reth v2.2 alignment:** Spencer aligns at the internal v2.2 payload-loop level; Arshan aligns at the nodebuilder extension-point level. For future merge cost, Arshan's latest split is cleaner if behavior is proven equivalent.
  - **Node-builder alignment:** Arshan is now closer to the target architecture: a custom `PayloadBuilderBuilder` selected by `RitualNode`. Spencer's conceptual placement is right, but should be extracted out of Reth payload to reach the same architecture.
  - Important review question: compare payload traces for scheduled-only, async-only, fulfilled async, delivery, sequencing-rights, and mixed blocks.
- **7. Block verification and consensus gates: both use v2.2 engine-tree surfaces; Arshan's latest commits improve verifier policy/context.**
  - Old `dev` had broad fail-closed verifier behavior for async commitments, settlements, SPC, scheduled constraints, nonce locks, censorship, and sequencing rights. Spencer ports this into `crates/ritual-block-verification/src/lib.rs`. Arshan keeps it under `crates/ritual/block-verification` and the latest commits centralize `SKIP_VALIDITY_CONSTRAINTS` policy and set verification context for engine payloads.
  - **Reth v2.2 alignment:** both branches carry upstream engine-tree and payload-validator surfaces. Arshan's latest context work is directly relevant to engine payload validation; Spencer has broader builder/verifier behavioral wiring.
  - **Node-builder alignment:** the clean end state is a Ritual payload validator / engine validator add-on plus shared verifier crate, rather than verifier logic scattered through engine tree or generic consensus code.
  - Important review question: run invalid-block fixtures per verifier phase and one builder-produced block replay/import test to prove builder and verifier agree.
- *8. RPC and operator surfaces: Arshan restores txpool and nodebuilder Ritual RPC; Spencer preserves a broad `ritual_` operator surface.**
  - Old `dev` exposed both txpool metadata and `ritual_`* operator methods for async result submission, delivery result submission, heartbeat state, block-build traces, scheduled incidents, and simulation traces. Arshan's latest Reth branch restores `txpool_scheduledContent` and `txpool_asyncContent`-style txpool extensions, while latest nodebuilder exposes Ritual add-ons for verifier policy, async submission, heartbeat state, block-build trace, and scheduled incidents. Spencer includes a broader `ethereum/node/src/ritual_async_rpc.rs` module and `rpc-api/src/ritual.rs` methods for async submission, delivery submission, heartbeat, block-build trace, and scheduled incidents.
  - **Reth v2.2 alignment:** txpool RPC extensions require `TransactionPool` trait additions; broader operator APIs fit better as node RPC add-ons.
  - **Node-builder alignment:** both now have node-level RPC/add-on wiring to inspect. Arshan's latest nodebuilder branch is no longer just txpool visibility; verify method parity rather than assuming the broader surface is missing.
  - Important review question: run JSON-RPC smoke tests for txpool scheduled/async content and every `ritual_`* method the product expects.
- **9. Chainspec, genesis, and config: both touch upstream config surfaces; Spencer appears broader, Arshan is actively restoring specific gaps.**
  - Old `dev` preserved Ritual `extra_data`, `base_block_reward`, max block size, eager-payload behavior, and retry/sync policy. Spencer has visible support for extra data, max block size config, base block reward, and payload config knobs. Arshan's latest commits restore payload identity extra data and expose genesis reward fallback, which directly addresses two important map items.
  - **Reth v2.2 alignment:** these belong in chainspec/genesis parsing, payload builder config, and EVM execution reward logic. They are upstream-surface changes, not node-builder-only changes.
  - **Node-builder alignment:** a custom Ritual node type can own chain spec and config defaults, but persisted genesis/header semantics must remain in shared chain and execution code.
  - Important review question: parse Ritual genesis, build/import a block with large/CL-provided `extra_data`, verify payload identity, and verify base-block reward application.
- **10. Fees, registries, and product economics: Arshan keeps support crates namespaced and consumes them from nodebuilder; Spencer connects them from Reth payload/verifier.**
  - Old `dev` used async-fee, registry, model, wallet, and TEE capability helpers across RPC, payload, and verifier. Arshan keeps substantial support code under `crates/ritual/async-fees`, `crates/ritual/async-registry`, and `crates/ritual/tee-registry-reader`, and latest nodebuilder payload/verifier/RPC paths consume those helpers. Spencer keeps flat support crates and calls them from `ethereum/payload/src/ritual_async.rs` and `ritual-block-verification`.
  - **Reth v2.2 alignment:** pricing and registry logic should remain in support crates; Reth integration points should only call deterministic helpers from payload/verifier/RPC.
  - **Node-builder alignment:** Arshan is closer to the ideal extraction shape if the nodebuilder calls are behaviorally complete. Spencer has more behavior in Reth, but less separation.
  - Important review question: run fee formula tests plus one end-to-end async settlement for executor, commitment, inclusion, Phase 2 delivery, model registry, wallet lock, and capability matching.
- **11. Observability and agent guardrails: Arshan now emits nodebuilder payload metrics/traces; Spencer preserves substantial in-Reth payload/operator observability.**
  - Old `dev` had block-build traces, demotion/cap observations, scheduled incidents, txpool async/scheduled metrics, and trace routing. Spencer carries block-build trace, scheduled incident, deadline, demotion, and RPC operator visibility. Arshan has restored txpool visibility and latest nodebuilder payload code emits Ritual payload metrics, phase traces, scheduled incident capture, heartbeat revival selection logs, stale scheduled cleanup logs, and async/scheduled build traces.
  - **Reth v2.2 alignment:** observability should attach through metrics, tracing, and RPC add-ons; it should not force large payload rewrites except where the payload code is the source of trace facts.
  - **Node-builder alignment:** a Ritual node add-on should expose trace/RPC views, while payload components emit structured events. Arshan's latest shape is closer to that separation; Spencer still has more observability tied directly to Reth payload/RPC code.
  - Important review question: enable trace envs, build a capped/demoted block with scheduled and async work, then verify metric names, trace entries, txpool views, and RPC debug responses.
